# Repository Guidelines

## Project Structure & Module Organization

This is a Swift Package for one CLI executable, `iOSFeedBot`.

- `Package.swift` defines the executable target and the FeedKit dependency.
- `Sources/iOSFeedBot/` contains all production code.
- `Sources/iOSFeedBot/main.swift` wires config validation, feed fetching, post generation, and Telegram publishing.
- Service files are grouped by responsibility, such as `FeedService.swift`, `AIService.swift`, and `TelegramService.swift`.
- `Models.swift` holds shared data models, while `Config.swift` reads environment-based configuration.
- `Tests/iOSFeedBotTests/` contains XCTest coverage for parsing and service behavior.
- `.build/` is generated by SwiftPM and should not be edited or committed.

## Product Requirements

The product is a Telegram bot that publishes one iOS development article per day. Each run should fetch recent iOS-related posts, select the strongest article from the last 24 hours, format it for Telegram, and post it to the configured channel. Scheduling is external, for example cron on a VPS.

## Build, Test, and Development Commands

- `swift build` builds the package in debug mode.
- `swift build -c release` creates `.build/release/iOSFeedBot`.
- `swift run iOSFeedBot` builds and runs the executable in one step.
- `swift test` runs the XCTest suite.

To run locally, set required environment variables first:

```bash
export OPENAI_API_KEY=your_key
export TELEGRAM_BOT_TOKEN=your_token
export TELEGRAM_CHANNEL_ID=@your_channel
swift run iOSFeedBot
```

## Coding Style & Naming Conventions

Use Swift 5.9 conventions with four-space indentation. Prefer service types named by responsibility, for example `FeedService`. Use `UpperCamelCase` for types and `lowerCamelCase` for functions, properties, and variables. Keep async network code explicit with `async`/`await`, and preserve Linux compatibility imports when using networking:

```swift
#if canImport(FoundationNetworking)
import FoundationNetworking
#endif
```

Avoid broad refactors when changing a single service. Keep configuration access centralized in `Config.swift`.

## Testing Guidelines

Use XCTest under `Tests/iOSFeedBotTests/`. Name test files after the unit, such as `FeedServiceTests.swift`.

Prefer focused tests for URL formatting, HTML escaping, config validation, feed parsing, and Telegram/OpenAI request construction. Mock network calls.

## Deployment

Production runs on the Hetzner VPS `89.167.48.211` as `root`. The checkout lives at `/root/ios-feed`, and cron runs `/root/ios_feed_run.sh` daily at 09:00 server time. That script owns production environment variables and executes `/root/ios-feed/.build/release/iOSFeedBot`.

Deploy only after pushing `main`:

```bash
ssh root@89.167.48.211
cd /root/ios-feed
git pull --ff-only origin main
swift test
swift build -c release
```

Do not run the bot manually during deployment unless an extra Telegram post is intended. Check `/root/ios_feed.log` for runtime output.

## Commit & Pull Request Guidelines

Recent history uses `feat:`, `fix:`, `docs:`, and `refactor:`. Follow that pattern with short, imperative summaries, for example `fix: handle missing atom links`.

Pull requests should include a brief description, configuration or deployment impact, test results or a note explaining missing tests, and relevant logs or screenshots for Telegram output changes. Never include API keys, bot tokens, or generated build artifacts.
